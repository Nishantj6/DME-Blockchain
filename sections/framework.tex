\section{Assurance Passport Framework}
\label{sec:framework}

This section defines the assurance passport as a W3C verifiable credential profile for engineering artifact attestation.
We specify the credential schema (Section~\ref{sec:fw-schema}), the lifecycle from issuance through verification (Section~\ref{sec:fw-lifecycle}), the on-chain anchoring mechanism (Section~\ref{sec:fw-anchoring}), how the proof predicates defined in Section~\ref{sec:proof-predicates} are bundled into a single credential (Section~\ref{sec:fw-bundling}), and how configuration baselines are attested (Section~\ref{sec:fw-baseline}).

\subsection{Credential Schema}
\label{sec:fw-schema}

An assurance passport is a domain-specific VC profile, analogous to Open Badges 3.0 for educational achievements~\cite{open_badges3} and the UNTP Digital Product Passport for supply chain sustainability~\cite{untp_dpp}.
It specializes the W3C Verifiable Credentials Data Model v2.0~\cite{w3c_vc} for engineering artifact attestation.

Formally, an assurance passport $\mathsf{AP}$ for artifact $a_i^{v}$ is a tuple:
\begin{equation}
\label{eq:ap-tuple}
\mathsf{AP}(a_i^{v}) = \left(\mathsf{meta},\; \mathsf{claims},\; \mathsf{proof}\right)
\end{equation}
where each component maps to the W3C VC data model:

\textbf{Metadata} $\mathsf{meta}$:
\begin{itemize}
    \item \texttt{id}: Unique credential identifier (DID-resolvable URI).
    \item \texttt{type}: \texttt{["VerifiableCredential", "AssurancePassport"]}.
    \item \texttt{issuer}: DID of the issuing stakeholder $S_i$.
    \item \texttt{issuanceDate}: Timestamp $t$ of credential creation.
    \item \texttt{expirationDate}: Optional validity horizon (e.g., tied to configuration baseline reviews).
    \item \texttt{credentialStatus}: Reference to the on-chain credential status registry (see Section~\ref{sec:fw-anchoring}).
\end{itemize}

\textbf{Claims} $\mathsf{claims}$:
\begin{itemize}
    \item \texttt{subject}: Artifact identifier and version $(a_i, v)$.
    \item \texttt{commitment}: On-chain commitment $c_i^{v} = H(a_i^{v})$.
    \item \texttt{predicates}: A set of attestation types from $\{\text{schema}, \text{version}, \text{origin}, \text{lineage}\}$ (see Section~\ref{sec:proof-predicates}).
    \item \texttt{schemaRef}: Identifier of the target schema $S$ (for schema compliance predicates).
    \item \texttt{parentCommitment}: Commitment $c_i^{v-1}$ to the predecessor version (for version validity predicates).
    \item \texttt{toolchain}: Identifier of the tool $T \in \mathcal{T}_{\text{approved}}$ used to produce the artifact. Only the tool identifier is recorded, not the tool binary.
\end{itemize}

\textbf{Proof} $\mathsf{proof}$:
\begin{itemize}
    \item \texttt{type}: \texttt{"Groth16Proof2024"}, an author-defined proof suite identifier indicating Groth16 zk-SNARKs on BN254.
    \item \texttt{proofValue}: The bundled ZK proof $\boldsymbol{\pi}$ (see Section~\ref{sec:fw-bundling}).
    \item \texttt{verificationMethod}: On-chain address of the verification key registry.
    \item \texttt{anchorTx}: Transaction hash anchoring the commitment on the consortium blockchain.
\end{itemize}

\subsection{Passport Lifecycle}
\label{sec:fw-lifecycle}

The assurance passport lifecycle consists of four phases, illustrated in Fig.~\ref{fig:lifecycle}.

\textbf{Phase 1: Artifact Commitment.}
When stakeholder $S_i$ completes artifact $a_i^{v}$ using tool $T$, the system computes the commitment $c_i^{v} = H(a_i^{v})$ and submits a transaction to the consortium blockchain recording $(c_i^{v}, v, S_i, T, t)$.
The blockchain provides an immutable timestamp $t$ and ordering guarantee.
The artifact content remains off-chain, held by $S_i$.

\textbf{Phase 2: Proof Generation.}
The stakeholder's local proof generation node---a dedicated computation environment co-located with the stakeholder's engineering tools (see Section~\ref{sec:architecture})---constructs the ZK proofs for the applicable predicates.
For each predicate $P_k$ in the passport's claim set:
\begin{equation}
\label{eq:proof-gen}
\pi_k \leftarrow \textsf{Prove}(\mathsf{pk}_k,\; x_k,\; w_k)
\end{equation}
where $\mathsf{pk}_k$ is the proving key for predicate $k$, $x_k$ is the public input (e.g., commitments, schema identifiers), and $w_k$ is the private witness (e.g., artifact content, credentials).
Proof generation is the computationally expensive step and occurs on the stakeholder's local hardware.

\textbf{Phase 3: Credential Issuance.}
The stakeholder assembles the assurance passport $\mathsf{AP}(a_i^{v})$ by combining the metadata, claims, and proof components.
The credential is self-issued: the stakeholder who produced the artifact is the issuer.
Self-issuance is justified because the trust anchor is not the issuer's identity but the combination of (a)~the ZK proof's mathematical soundness, (b)~the on-chain commitment's immutability, and (c)~the authorized origin predicate $\pi_{\text{origin}}$, which cryptographically binds the proof to a consortium-permissioned Ethereum account.
Specifically, the commitment transaction in Phase~1 is signed by $S_i$'s consortium-permissioned account, and the origin predicate proves that this account belongs to $\mathcal{S}_{\text{auth}}(a)$; together, these bind the issuer identity to the proof without relying on the VC metadata alone.
Note that if $\pi_{\text{origin}}$ is omitted from the passport's predicate set, the identity binding relies solely on the permissioned blockchain's access control---the credential's \texttt{issuer} field is not independently cryptographically bound.

In defense acquisition contexts, self-issued passports attest to structural properties (schema compliance, provenance, lineage) but do not replace third-party acceptance.
A designated authority (e.g., a program management office or IV\&V agent) may issue an \emph{endorsement credential}---a separate VC that references the self-issued passport and attests to programmatic acceptance.
This two-layer model separates cryptographic assurance (self-issued) from governance approval (authority-endorsed).

The issued credential is stored in the stakeholder's credential wallet and can be presented to any verifier.

\textbf{Phase 4: Verification.}
A verifier (e.g., a program manager, auditor, or downstream stakeholder) receives the assurance passport and checks:
\begin{enumerate}
    \item \textbf{On-chain anchor}: The commitment $c_i^{v}$ referenced in the claims matches a commitment recorded on-chain at the claimed time $t$.
    \item \textbf{Public input consistency}: The commitment $c_i^{v}$ used as public input is identical across all predicate proofs in the bundle, ensuring they attest to the same artifact.
    \item \textbf{ZK proof validity}: For each predicate proof $\pi_k$:
    \begin{equation}
    \label{eq:verify}
    \textsf{Verify}(\mathsf{vk}_k,\; x_k,\; \pi_k) \stackrel{?}{=} 1
    \end{equation}
    \item \textbf{Credential status}: The credential has not been revoked (checked against the on-chain status registry).
    \item \textbf{Temporal validity}: The current time is within the credential's validity window.
\end{enumerate}
Verification requires only the public inputs, verification keys, and blockchain access---the verifier never sees the artifact content.

\subsection{On-Chain Anchoring}
\label{sec:fw-anchoring}

The consortium blockchain serves three functions in the assurance passport framework:

\textbf{1. Commitment Registry.}
A smart contract maintains a mapping from artifact identifiers to their commitment history:
\begin{equation}
\label{eq:registry}
\mathsf{Registry}: (a_i, v) \mapsto (c_i^{v}, S_i, t, \mathsf{txHash})
\end{equation}
Each entry records the commitment $c_i^{v} = H(a_i^{v})$, the contributing stakeholder $S_i$, the block timestamp $t$, and the anchoring transaction hash.
The registry is append-only: new versions can be added but existing entries cannot be modified, providing tamper evidence (Security Goal~1) and temporal integrity (Security Goal~2).
Write access is enforced by the smart contract: only a stakeholder $S_i \in \mathcal{S}_{\text{auth}}(a)$ (authenticated via their consortium-permissioned Ethereum account) can commit a new version for artifact $a_i$.
This prevents front-running attacks where a malicious stakeholder registers a commitment for an artifact they do not own.

\textbf{2. Verification Key Registry.}
The verification keys $\mathsf{vk}_k$ for each proof predicate are stored on-chain, ensuring that all verifiers use the same keys derived from the trusted setup.
Verification key entries are immutable once written; updates (e.g., after a new trusted setup ceremony) require a governance transaction approved by a quorum of consortium members.
This prevents a malicious party from substituting a verification key that would accept forged proofs.

\textbf{3. Credential Status Registry.}\label{sec:fw-revocation}
An on-chain registry tracks the revocation status of issued passports, following the W3C Bitstring Status List pattern.
A passport may be revoked when:
\begin{itemize}
    \item The underlying artifact is superseded and the stakeholder wishes to invalidate prior attestations.
    \item A security incident compromises the stakeholder's signing key.
    \item A consortium governance decision invalidates a class of credentials (e.g., all passports issued using a deprecated schema version).
\end{itemize}
Revocation is a consortium-governed action.
The smart contract enforces that only the original issuer or a designated consortium authority (e.g., the program management office) can revoke a passport.
Each revocation transaction is recorded immutably for auditability.

\subsection{Proof Bundling}
\label{sec:fw-bundling}

An assurance passport may attest to multiple properties simultaneously.
Rather than issuing separate credentials for each predicate, we bundle the proofs into a single credential.
The proof bundle $\boldsymbol{\pi}$ in the passport's proof field is the concatenation of individual predicate proofs, each independently verifiable:
\begin{equation}
\label{eq:bundling}
\boldsymbol{\pi} = (\pi_{\text{schema}},\; \pi_{\text{version}},\; \pi_{\text{origin}},\; \pi_{\text{lineage}})
\end{equation}

Each component proof is independently verifiable: a verifier can check any subset of predicates without processing the full bundle.
This design supports incremental attestation---a stakeholder can issue an initial passport with only $\pi_{\text{schema}}$ and $\pi_{\text{origin}}$, then update it with $\pi_{\text{version}}$ when a new version is committed.
Note that this is concatenation-based bundling, not recursive proof composition (e.g., SnarkPack or Halo-style folding); verification cost scales linearly with the number of predicates ($4 \times 3 = 12$ pairing operations for a full bundle).

The total proof size for a full passport with all four predicates is $4 \times 128 = 512$ bytes compressed (four Groth16 proofs on BN254), or 1{,}024 bytes uncompressed as used by the EVM pairing precompiles.
Either representation is suitable for transmission in bandwidth-constrained DDIL environments.

\textbf{Binding guarantee.}
All four predicates share the artifact commitment $c_i^{v} = H(a_i^{v})$ as a common public input.
The verifier must confirm that the $c_i^{v}$ value is identical across all sub-proofs in the bundle (verification step~2 in Section~\ref{sec:fw-lifecycle}) and matches the on-chain registry.
An adversary cannot mix proofs from different artifacts because producing a valid proof for commitment $c_j^{v'} \neq c_i^{v}$ requires knowledge of the artifact $a_j^{v'}$ corresponding to $c_j^{v'}$; this follows from the \emph{knowledge-soundness} (extractability) of Groth16 under the Knowledge-of-Exponent assumption, which guarantees that a valid prover must ``know'' the witness.

\subsection{Configuration Baseline Attestation}
\label{sec:fw-baseline}

Individual assurance passports attest to single artifacts, but DME decisions are taken against a \emph{configuration baseline}---a coordinated set of artifact versions approved at a program milestone (e.g., System Requirements Review, Preliminary Design Review).
To attest to a baseline as a whole, we define a \emph{Baseline Credential}:
\begin{equation}
\label{eq:baseline}
\mathsf{BC}(d, t) = \left(\mathsf{meta}_d,\; \{c_1^{v_1}, \ldots, c_k^{v_k}\},\; \sigma_{\text{auth}}\right)
\end{equation}
where $\mathsf{meta}_d$ identifies the decision milestone, $\{c_1^{v_1}, \ldots, c_k^{v_k}\}$ is the set of artifact commitments constituting the baseline, and $\sigma_{\text{auth}}$ is a signature from the designated configuration authority (e.g., the program management office).
The baseline credential uses a conventional digital signature rather than a ZK proof because the commitment set $\{c_1^{v_1}, \ldots, c_k^{v_k}\}$ is already public on-chain; the baseline credential adds only the assertion that these specific commitments were jointly approved at milestone $d$.
A verifier can confirm that each individual passport in the baseline set was valid at time $t$ by checking the corresponding commitment registry entries and ZK proofs.

The baseline credential binds the individual passports to a decision point, enabling post-hoc audit: ``these exact artifact versions constituted the ASOT when decision $d$ was taken.''
This directly supports Temporal Integrity (Security Goal~2) at the baseline level, not just the individual artifact level.

\textbf{Scope of current predicates.}
The four proof predicates (schema compliance, version validity, authorized origin, derivation lineage) address \emph{structural} integrity and provenance.
They do not attest to Verification \& Validation (V\&V) status or accreditation for intended use, which are programmatic judgments governed by DoD policy (e.g., DoDI~5000.61 for M\&S VV\&A).
V\&V attestation could be added as a fifth predicate in future work, or modeled as an endorsement credential issued by an IV\&V agent.
We discuss this extension in Section~\ref{sec:discussion}.

\subsection{Security Analysis}
\label{sec:fw-security}

We now argue that the assurance passport framework satisfies the security goals defined in Section~\ref{sec:security-goals}.

\textbf{Tamper Evidence (Goal~1).}
An adversary attempting to substitute $a_i' \neq a_i^{v}$ after commitment must either (a)~find a collision $H(a_i') = H(a_i^{v})$, which contradicts the collision resistance of $H$, or (b)~modify the on-chain registry, which requires corrupting more than $\lfloor (n-1)/3 \rfloor$ validators (contradicting the honest-supermajority assumption).

\textbf{Temporal Integrity (Goal~2).}
The blockchain's consensus mechanism provides a total ordering of transactions with finality.
The commitment registry maps each $(a_i, v)$ to a unique block timestamp, enabling reconstruction of the ASOT state at any historical time $t$.
Under IBFT/QBFT, finalized blocks cannot be reverted without violating the Byzantine fault tolerance bound.

\textbf{Change Accountability (Goal~3).}
Each commitment transaction records the stakeholder identity $S_i$ and is signed with $S_i$'s private key.
The version validity predicate ($\pi_{\text{version}}$) further attests that the transition from $a_i^{v}$ to $a_i^{v+1}$ follows the approved update rules, providing a verifiable change trail.

\textbf{Authorized Provenance (Goal~4).}
The authorized origin predicate ($\pi_{\text{origin}}$) proves set membership ($S_i \in \mathcal{S}_{\text{auth}}(a)$ and $T \in \mathcal{T}_{\text{approved}}$) via Merkle inclusion proofs without revealing the stakeholder's credentials or tool identity to the verifier.
Soundness of the zk-SNARK ensures that only a prover with valid credentials can produce a valid proof.

\textbf{Selective Disclosure (Goal~5).}
By construction, the verifier receives only the public inputs $(c_i^{v}, S, \mathsf{root}_S, \mathsf{root}_T, \ldots)$ and the proofs $\boldsymbol{\pi}$.
The zero-knowledge property of Groth16 guarantees that no information about the witness (artifact content $a_i^{v}$, stakeholder credential, tool identity, Merkle paths) is leaked beyond what is implied by the truth of the predicates.

\section{Evaluation}
\label{sec:evaluation}

We evaluate the assurance passport system on the edge hardware testbed described in Section~\ref{sec:arch-hardware}.
The evaluation addresses three questions:
(1)~Is proof generation feasible on resource-constrained edge hardware?
(2)~How do consensus protocols affect end-to-end latency and throughput?
(3)~What are the on-chain costs of commitment anchoring and proof verification?

\subsection{Experimental Setup}
\label{sec:eval-setup}

\textbf{Hardware.}
The testbed comprises four NVIDIA Jetson Nano 4~GB boards connected via Gigabit Ethernet, each running Ubuntu 18.04 (JetPack 4.6) with GoQuorum v23.4 as the blockchain client.
Circuit compilation and trusted setup are performed offline on an Apple M4 Max laptop.

\textbf{Software.}
ZK circuits are implemented in Circom 2.1~\cite{belles2023circom} and compiled to R1CS.
Proof generation uses SnarkJS with the Groth16 prover on BN254.
Smart contracts are deployed using Hardhat with Solidity 0.8.x.
The on-chain Poseidon library uses an optimized Solidity implementation with arity-4, matching the in-circuit parameterization.

\textbf{Workload.}
We use synthetic SysML-like artifacts serialized to canonical JSON, ranging from 1~KB to 8~KB (32 to 256 BN254 field elements).
Each experiment is repeated 30 times; we report the median with interquartile range (IQR).

\textbf{Consensus configurations.}
We benchmark three consensus protocols on the same four-node testbed: IBFT (BFT, $f \leq 1$), QBFT (BFT, $f \leq 1$), and Raft (CFT, $f \leq 1$).
Raft serves as a performance upper bound since it eliminates the multi-round Byzantine agreement overhead.

Each Groth16 verification invokes the BN254 \texttt{ecPairing} precompile (address 0x08), which dominates the gas cost.
Under the post-EIP-1108 gas schedule (Istanbul fork, included in GoQuorum v23.4), the \texttt{ecPairing} precompile costs $34{,}000 + 45{,}000 \times k$ gas for $k$ input pairs.
The actual gas consumed on the testbed will be reported when TBD values are filled; the figures below use the EIP-1108 formula as a reference.
A standard Groth16 verification uses 4 pairing pairs (proof elements $A, B, C$ plus the public-input accumulator), costing approximately $34{,}000 + 4 \times 45{,}000 = 214{,}000$ gas, plus additional gas for \texttt{ecMul}/\texttt{ecAdd} operations during public-input accumulation.
For the full bundle with four independent predicate verifications, the pairing cost alone is approximately $4 \times 214{,}000 \approx 856{,}000$ gas.
As noted in Section~\ref{sec:arch-contracts}, GoQuorum operates with zero gas price; these figures represent computational metering, not economic cost.
For comparison, the equivalent verification on Ethereum mainnet at 30~gwei and \$3{,}000/ETH would cost approximately \$0.02 per single predicate or \$0.08 for the full bundle---modest given the cryptographic guarantees provided.

\textbf{On-chain verification latency.}
On-chain verification is fast relative to proof generation.
Our prior work~\cite{jagannath2025zksnark} reported 61.50~s end-to-end for on-chain verification on a public testnet; this figure is dominated by transaction confirmation latency (block time and finality), not the verification computation itself, which takes milliseconds.
On the local consortium chain, verification completes within a single block time, as consensus latency dominates.

\subsection{Discussion of Results}
\label{sec:eval-discussion}

The evaluation demonstrates that the assurance passport system is feasible on edge hardware, with proof generation as the performance bottleneck.
Key observations:

\begin{enumerate}
    \item \textbf{Proof generation is the bottleneck.} On the Jetson Nano, proof generation dominates end-to-end latency. However, proof generation is a local, asynchronous operation: it does not block the blockchain or other stakeholders, and can be performed in batch during periods of connectivity.

    \item \textbf{Consensus choice matters for throughput, not latency.} For individual passport issuance, the consensus protocol has minimal impact on perceived latency (proof generation dominates). Under concurrent load, QBFT provides a practical balance of Byzantine fault tolerance and throughput.

    \item \textbf{On-chain verification is lightweight.} Groth16 verification via BN254 precompiles is computationally inexpensive relative to proof generation, confirming the asymmetry that makes ZK proofs suitable for verifier-constrained environments.

    \item \textbf{DDIL suitability.} The system operates without external connectivity once the consortium chain is initialized. Proof generation is fully local, and verification requires only intra-consortium communication. The 512-byte proof bundle is suitable for bandwidth-constrained links.
\end{enumerate}
